# Tools for Agentic Coding

Over the past year I have been building and using AI coding agents extensively.
Along the way I kept running into the same friction points: agents that don't
know enough about the project, repositories that need to share files across
boundaries, and documentation that is painful to maintain. So I built tools to
solve each one.

> [!tip] All three of these tools are open source and designed to work together.
> They are small, opinionated, and assume you are using git.

## context-db

[context-db](https://github.com/cart0113/context-db) is a hierarchical Markdown
knowledge base designed for LLM agents. The idea is simple: instead of forcing
an agent to read your entire codebase to understand how things work, you write
short documents with YAML frontmatter descriptions and organize them in folders.

An agent reads the table of contents, sees one-line descriptions for each
document, and only fetches what is relevant to its current task. This keeps
context usage efficient and prevents the agent from being overwhelmed.

The key features:

- **On-demand tables of contents** generated by `bin/show_toc.sh` — no manual
  index maintenance
- **YAML frontmatter** with `description` fields that agents use to decide
  relevance before reading
- **Folder hierarchy** for organizing knowledge by topic (architecture, coding
  standards, deployment, etc.)
- **Cross-project sharing** via git-sync — the same context-db documents can be
  symlinked into multiple repositories

I use context-db in every project now. It has become the primary way I teach
agents about project architecture, data models, and coding conventions.

## git-sync

[git-sync](https://github.com/cart0113/git-sync) is a bash tool for composing
independent git repositories into a pseudo-monorepo. Think of it as a simpler,
more transparent alternative to `git submodule`.

You define a `.git-sync.yaml` file that declares which external repos to pull in
and where they should live:

```yaml
repos:
  context-db:
    url: https://github.com/cart0113/context-db.git
    path: bin
    mode: checkout-commit
    commit: abc123
```

Then `git-sync pull` checks out the specified files into your project. Two sync
modes are available:

- **checkout-commit** — pins to an exact commit hash for reproducibility
- **update-branch** — tracks a branch and pulls the latest

I built this because `git submodule` has confusing edge cases that trip up both
humans and AI agents. git-sync is explicit: you can see exactly what version of
what repo is checked out, and the YAML file makes it obvious.

## bruha

[bruha](https://github.com/cart0113/bruha) is a set of extensions for
[docsify](https://docsify.js.org/) that I use for all my documentation sites. It
is not a fork of docsify — docsify loads from a CDN — but rather a layer of CSS
themes, JavaScript plugins, and a YAML-driven configuration system on top.

The documentation is live at
[cart0113.github.io/bruha](https://cart0113.github.io/bruha/).

What it provides:

- **JSON configuration** — one file (`bruha.json`) controls everything: themes,
  sidebar behavior, navigation style, code highlighting
- **Auto-generated sidebars** — the sidebar is built from the filesystem using
  `_order.md` files for sequencing, so you never hand-edit `_sidebar.md`
- **Multiple visual styles** — code-one for technical documentation, blog for
  editorial content (like this site)
- **Color themes with dark mode** — each style has its own set of themes with
  light and dark variants and a theme picker UI
- **Callout boxes** — styled alert boxes using `> [!tip]` syntax

This blog itself is powered by bruha using the blog style.

## How They Work Together

In a typical project, I set up all three:

1. **context-db** provides the knowledge base that agents read at the start of
   every session
2. **git-sync** pulls context-db's tooling (the `bin/show_toc.sh` script) into
   the project
3. **bruha** powers the project's documentation site, with context-db documents
   informing the agent about how bruha works

The result is a workflow where an AI agent can pick up any of my projects, read
the context-db to understand the architecture, and start contributing
effectively with minimal ramp-up.
