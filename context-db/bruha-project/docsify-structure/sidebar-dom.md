# Sidebar DOM Structure

## Source: \_sidebar.md

Docsify loads `_sidebar.md` asynchronously and renders it as a `<ul>` tree
inside `.sidebar-nav`. The sidebar file is generated by the Python sidebar
builder from the filesystem using `_order.md` files (see `sidebar_builder.md`).

Example `_sidebar.md`:

```markdown
- **Overview**
  - [Overview](overview/overview.md)
- **Configuration**
  - [Configuration Reference](configuration/config-reference.md)
```

## Rendered DOM

Docsify renders this as nested `<ul>/<li>` elements:

```
.sidebar-nav
  └── <ul>                              ← root ul
      ├── <li class="ext-folder">       ← bold text = folder
      │   ├── <strong>Overview</strong> ← folder header (no <a>)
      │   └── <ul>
      │       └── <li class="sb-page-link sb-active-page sb-has-sections">
      │           ├── <a href="#/overview/overview">Overview</a>
      │           └── <ul class="app-sub-sidebar">   ← DOCSIFY-GENERATED
      │               ├── <li><a href="...?id=how-it-works">How It Works</a></li>
      │               └── <li><a href="...?id=quick-start">Quick Start</a></li>
      └── <li class="ext-folder">
          ├── <strong>Configuration</strong>
          └── ...
```

Key distinction:

- **Static sidebar items** come from `_sidebar.md` — `<a>` links to pages
- **`app-sub-sidebar`** is injected by docsify from the current page's headings
  (H2, H3, etc.)

## app-sub-sidebar

Docsify generates the `app-sub-sidebar` `<ul>` from the active page's headings
and injects it as a child of the active page's `<li>`. It is:

- **Only present for the active page** — other pages don't have it
- **Rebuilt on every page navigation** — docsify removes the old one and creates
  a new one
- **Controlled by `subMaxLevel`** in `index.html` docsify config (set to
  `document_header_depth` from YAML)
- Contains `?id=` links for scrolling to sections within the page

### H3+ nesting: sibling `<ul>`, not child of `<li>`

When `subMaxLevel >= 3`, docsify nests H3 headings in a **sibling `<ul>`** of
the H2 `<li>` elements — NOT as a child `<ul>` inside the H2's `<li>`:

```
<ul class="app-sub-sidebar">
  <li><a>H2: Example Config</a></li>
  <li><a>H2: Keys</a></li>
  <ul class="app-sub-sidebar">        ← sibling <ul>, NOT inside Keys <li>!
    <li><a>H3: theme_name</a></li>
    <li><a>H3: theme_controls</a></li>
  </ul>
  <li><a>H2: Another Header</a></li>
</ul>
```

This is invalid HTML (`<ul>` can only contain `<li>`), but it's what docsify
generates. **Any JS that walks the sub-sidebar must account for this** —
querying `li.querySelector(':scope > ul')` will NOT find H3 items. Instead, walk
`ul.querySelectorAll(':scope > ul')` to find sibling `<ul>` children.

The inner `<ul>` also carries the `app-sub-sidebar` class.

## Critical: Re-render Behavior

**Letting a click propagate to docsify on the active page link will trigger a
re-render that destroys the `app-sub-sidebar`.** This is why
`collapsible-folders.js` must `preventDefault()` and
`stopImmediatePropagation()` on all active-page link clicks, and handle
navigation manually via `history.pushState` + `hashchange` dispatch.

When docsify re-renders:

1. Sidebar DOM is rebuilt from `_sidebar.md`
2. Page content is re-rendered
3. New `app-sub-sidebar` is generated from headings
4. `doneEach` hooks fire after rendering is complete

The problem: between step 1 and step 3, the sub-sidebar children are gone. If
any click handler or observer runs during this window, state can be lost.

## Top Nav Mode

When `top_level_folders_as_top_control: true`:

- Top-level `<li class="ext-folder">` items become hidden tabs
- `top-nav.js` adds `ext-top-hidden` class to non-active folders
- Folder `<strong>` headers are hidden (`display: none`) since the tab replaces
  them
- Only the active folder's children are visible in the sidebar
- `collapsible-folders.js` skips top-level folders (checks
  `folder.parentElement === rootUl`)
